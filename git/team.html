チーム開発を学ぶ

複数人で1つのリポジトリを共有すると、「誰かの変更で自分の作業が消える」「main が壊れる」といった事故が起きやすくなります。 それを防ぐのが ブランチ → Pull Request（PR）→ レビュー → マージ という決まったリズムです。 ここでは、その1サイクルと注意点を図で整理します。ブランチそのものが不安なら、先に ブランチを学ぶへ。コマンドの動きは シミュレーターで試せます。

① まず守る4つの基本原則

細かいコマンドより先に、この4つだけは全員で守ります。ほとんどの事故はここで防げます。

守ること

  • main には直接 push しない（必ずPR経由で反映）
  • 作業は必ずブランチを切ってから始める
  • 作業前に最新の main を取り込む（古い状態から分岐しない）
  • 1つのPRは1つの目的に絞る（レビューしやすく、戻しやすい）

なぜ？

  • main は「いつでも動く本番の流れ」。壊すとチーム全員が止まる
  • ブランチなら、未完成でも他の人に影響しない
  • 古い main からの分岐はコンフリクトの温床
  • 大きなPRはレビュー不能・問題の切り分け不能になる
このサイト自体が実例です： Git Visual Trainer も main に直接ではなく作業ブランチで作られ、確認できてから main に合流しています。 詳しくは ブランチを学ぶの④を参照。

② 開発1サイクルの全体像

チーム開発は、下の流れをぐるぐる繰り返します。左の main から枝分かれして作業し、 PRとレビューを通って、また main に戻ってくる——この1周が基本単位です。

main 側（共有の流れ）
  1. 最新の main を取得（git pull）
  2. ここで作業ブランチを切る（git switch -c feature/xxx）
  3. レビュー中も main は他の人のPRで進むことがある
  4. PRをマージして main に反映
分岐 → ← マージ
作業ブランチ側（自分の作業）
  1. 作業して commit → push
  2. GitHubで PRを作成・レビュー依頼
  3. 指摘に対応（同じブランチに追記push）
  4. マージ前に main を取り込み → 承認 → マージ後ブランチ削除
1行でいうと： main最新を取得 → ブランチを切る → 作業・commit → push → PR作成 → レビュー → 修正 → main取り込み → マージ → ブランチ削除

③ 作業を始める前（ブランチを切る）

毎回この3行から始めます。「最新の main から枝分かれする」のが肝心です。

git checkout main git pull origin main # mainを最新にする git switch -c feature/xxx # 作業ブランチを切る

ブランチ名の付け方

ブランチ名の接頭辞と用途
接頭辞用途例
feature/新機能feature/login-form
fix/バグ修正fix/card-layout
refactor/リファクタrefactor/storage-layer
docs/ドキュメントdocs/update-readme
秘密情報・不要ファイルはコミットしない： パスワード・APIキー・.env や、自動生成される node_modules/ などは .gitignore に書いて追跡対象から外します。 一度コミットしてpushした秘密情報は履歴に残り続けるため、最初に .gitignore を整えるのが安全です。

④ Pull Request（PR）とは？良い書き方

PRは 「この変更を main に取り込んでください」という提案＋レビューの場です。 GitHub上で feature/xxx → main のPRを開き、レビュアーを指定して依頼します。 いきなりマージせず、人の目を通してから main に入れるのが狙いです。

PR説明に書くこと

  • 何を変えたか
  • なぜ変えたか
  • 確認方法 / スクリーンショット（UI変更なら）
  • 関連Issueがあれば Closes #123 で紐付け

PRを上手に使うコツ

  • 小さく保つ（巨大PRはレビュー不能）
  • まだ途中なら Draft（下書き）PR にして「レビューはまだ」と示す
  • 依頼前に自分でも一度 diff を読み返す（self-review）
  • 議論はPRのコメントに残す（後から経緯を追える）
Issueの自動クローズ： PR説明に Closes #123 / Fixes #123 / Resolves #123 と書くと、 そのPRが main（デフォルトブランチ）にマージされたとき、対応するIssueが自動で閉じます。 ※デフォルトブランチ向けのPRでのみ有効です。

⑤ レビュー対応 → マージ前に main を取り込む

レビューで指摘があっても、PRを作り直す必要はありません。 同じブランチに追加コミットして push すれば、PRは自動で更新されます。

そして作業中に、他の人のPRが先に main にマージされていることがあります。 自分のPRをきれいにマージできる状態にするため、マージ前に main の最新を取り込みます。

git switch feature/xxx git pull origin main # mainの変更を取り込む # コンフリクトがあれば解消 → commit git push
指摘対応のたびに新しいPRを作らないこと。1つのPRを育てていくイメージです。 こうすると議論とコミットが1か所にまとまり、後から経緯を追えます。

⑥ マージ方法の選び方

承認（Approve）されたら、GitHubの Merge ボタンで main に反映します。 マージには3つの方式があり、main の履歴の見え方が変わります。

マージ方法の比較
方法特徴おすすめ度
Squash and merge ブランチのコミットを1つにまとめる。main履歴が「1機能＝1commit」で読みやすい ◎ 個人〜小規模チーム
Merge commit コミットを全部残し、マージ用commitを足す。履歴は詳細だが煩雑になりがち △
Rebase and merge マージcommitなしの一直線な履歴。hashが書き換わる ○
迷ったら Squash。 履歴の見やすさを優先します。チームで方式を1つに統一しておくと混乱しません。

⑦ マージ後の後片付け

マージしたら、役目を終えた作業ブランチを片付けて、次のサイクルに備えます。

git switch main git pull origin main # マージ結果を取り込む git branch -d feature/xxx # ローカルの作業ブランチを削除

GitHub上のブランチも、PR画面の 「Delete branch」 ボタンで削除しておくとリポジトリが散らかりません。

⑧ コンフリクトが起きたら（落ち着いて）

同じ場所を2人が別々に変えていると、Gitは「どちらを残すか自動で決められない」状態になります。これがコンフリクトです。 慌てず、次の手順で解消します。

  1. git pull origin main を実行すると、衝突箇所が <<<<<<< ======= >>>>>>> のマーカーで表示される
  2. どちらを残すか（または両方残すか）を手で決めて、マーカーを消す
  3. git add → git commit → git push
  4. 判断に迷うコードは勝手に解消せず、書いた本人に確認する
一番危ないのは「とりあえず片方を消す」： 消してはいけない他人のコードを消す事故につながります。中身が分からないときは必ず確認を。

⑨ やってはいけないこと

  • main に直接 commit / push しない
  • 巨大なPRを作らない — レビュー不能・コンフリクトの温床。小さく分ける
  • pullせずに作業を始めない — 古い main からの分岐は事故のもと
  • コンフリクトを適当に解消しない — 消してはいけないコードを消す事故
  • 共有ブランチ（main等）で git push --force しない
  • 他人のブランチに勝手に force push しない
  • コミットメッセージを fix update だけで済ませない
  • レビューは「動くか」だけでなく「読めるか／後で直せるか」も見る

⑩ コミットメッセージの目安（Conventional Commits）

接頭辞で種類を示すと、履歴がぐっと読みやすくなります。

Conventional Commits の接頭辞
接頭辞意味
feat:新機能
fix:バグ修正
refactor:動作を変えない内部改善
docs:ドキュメント
style:見た目・フォーマットのみ
test:テスト追加・修正

例: feat: タグバッジをサムネイルに重ねて表示
※「なぜそうしたか」が複雑なときは、本文（2行目以降）に書きます。

⑪ よく使うコマンド（チートシート）

git status # 現在の状態を確認 git branch # ブランチ一覧 git switch -c feature/xxx # ブランチ作成＆移動 git pull origin main # mainを取り込む git add . && git commit -m "" # 変更を記録 git push origin feature/xxx # リモートへ送る git log --oneline -10 # 履歴を簡潔に表示 git stash # 作業を一時退避 git stash pop # 退避した作業を戻す
