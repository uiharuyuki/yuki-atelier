ブランチを学ぶ

ブランチ（branch＝枝）は、Gitでいちばんつまずきやすい所です。 ここでは「ブランチとは何か」「main と作業ブランチの違い」「分岐と合流（merge）のしくみ」を、 図を見ながら整理します。コマンドの動きを試したくなったら、いつでも シミュレーターへ。

① ブランチとは？

ブランチは 「作業の枝分かれ」 です。 同じプロジェクトの中に“もう一つの作業ライン”を作り、 元の状態を壊さずに別の変更を進められるしくみです。

たとえば本番用の流れ（main）はそのままに、 新機能や試作だけを別ブランチで進め、できあがってから本番に合流させます。 失敗しても本番に影響しないのが大きな利点です。

イメージ： 1冊のノートをコピーして「下書き用」を作る感覚。下書きで好きに試して、 OKなら本物のノートに書き写す（=merge）、ダメなら捨てる、ができます。

② main と 作業ブランチ（feature）の違い

同じ「ブランチ」でも、役割が違います。使い分けが分かると一気に楽になります。

main ブランチ

  • プロジェクトの基準となる流れ
  • 基本的にいつでも動く状態を保つ
  • ここで直接いきなり実験はしない
  • 完成した変更を合流させていく場所

作業ブランチ（feature など）

  • main から枝分かれして作る
  • 新機能・修正・試作を気軽に試す場所
  • 失敗しても main に影響しない
  • 完成したら main に合流（merge）する
mainと作業ブランチの比較
観点main作業ブランチ
役割基準・本番試作・作業
安定性常に動く状態を保つ途中でも壊れてOK
作り方最初からあるgit switch -c 名前 で作る
終わり方ずっと残る合流したら役目を終える

③ 分岐と合流（merge）のしくみ

作業ブランチは main の途中から分岐し、 作業が終わったら main に合流（merge）します。 下の図は、その流れを上から下へ表したものです。

main ブランチ
  1. 最初のコミット
  2. ここで git switch -c feature（分岐）
  3. mainは必要に応じて進められる
  4. feature を合流（merge）して完成
分岐 → ← 合流
feature ブランチ
  1. 機能の作業 ①
  2. 機能の作業 ②
  3. 完成 → mainへ戻す

※ このサイトのシミュレーターは分岐・移動までを再現します。merge（合流）は今後の追加予定の機能です。

④ なぜ今このサイトは claude/… ブランチにあるの？

まさに②③の考え方の実例です。このサイトは main に直接作らず、 claude/git-visual-trainer-m7o9de という作業ブランチで作られています。

  • mainを汚さない：未完成の状態を本番（main）に混ぜずに作業できます。
  • 安全に試せる：このブランチで自由にプレビュー・修正できます。
  • 納得してから合流：内容を確認し、OKになったら main に合流（マージ）して公開します。
先ほどのGitHub Pagesプレビューも、この作業ブランチを指定して表示していました。 「作業ブランチで作って、確認できたらmainへ」——これが普段の開発の基本リズムです。

⑤ ブランチでよく使うコマンド

コマンド意味
git branchブランチ一覧を見る（* が今いる場所）
git switch ブランチ名既存ブランチへ移動する
git switch -c ブランチ名新しいブランチを作って同時に移動する（-c＝create）
⚠ よくある勘違い： git branch 名前 は作るだけで移動しません。 作って移動までしたいときは git switch -c 名前 を使います。
